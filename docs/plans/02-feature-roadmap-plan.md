## 背景 — 現状の機能はプロダクトの価値に対して限定的

Docutor が提供したい価値は「企業に眠る非構造文書を、人と AI エージェントが使える
構造化ナレッジ資産に変える」こと。しかし現状の実装は
**「1 ファイルを 1 回変換し、その場でレビューして、その場でダウンロードする」単発ツール**に
とどまっており、次のギャップがある。

- 変換した文書の一覧・履歴がなく、ページを離れると成果物に戻れない(URL を失えば終わり)。
- 対象ユーザーは日本企業なのに UI が英語のみ。
- 「AI エージェントが使えるナレッジ」を謳いながら、出力は単一の Markdown / ZIP のみで、
  RAG やエージェントに渡すためのチャンク・メタデータ形式がない。
- 企業文書は通常フォルダ単位で大量に存在する(`sample-inputs/` 参照)が、複数ファイルの
  一括変換ができない。
- レビューで「なぜ直したか」を残す手段がなく、再生成に人間の指示を与えられない。
- プロバイダ非依存を謳いながら実装は OpenAI のみ。

以下、優先度順(F-1 が最優先)。各機能は独立した PR 相当の単位で実装し、
`pnpm test` / `pnpm lint` / `pnpm build` を通してからコミットすること。

## F-1. ドキュメント履歴ダッシュボード

**価値**: 変換資産を「使い捨て」から「蓄積されるナレッジ」に変える最初の一歩。
現在はレビュー URL を失うと成果物へ戻れない。

**実装内容**:
- 新規ページ `/documents`: 変換済み文書の一覧(ファイル名、種別、変換日時、
  セクション数、accepted/pending/rejected 数、ステータス)。クリックで `/review/[id]` へ。
- データソースは 2 系統を合流させる:
  - クライアント文書: localStorage の `docutor:document:*` を列挙するヘルパを
    `client-document-store.ts` に追加(`listClientDocuments()`)。
  - サーバー文書: `GET /api/documents`(一覧)を新設。`runtime/documents/*/job.json` を
    走査して要約(id、ファイル名、status、更新日時)を返す。ReviewDocument 全体は返さない。
- 削除操作(localStorage キー削除 / `DELETE /api/documents/[id]` 新設)。
- アップロード画面ヘッダー(`app-header.tsx`)から `/documents` への導線を追加。

**受け入れ基準**: 変換 → ホームに戻る → `/documents` から同じレビューを再開できる。
削除した文書が一覧から消える。

## F-2. 原本ページビューア(全セクション共通のトレーサビリティ)

**価値**: レビューの本質は「原本と突き合わせて信頼できるか判断する」こと。
P0-1 で図セクションに原本画像が付くが、これを全セクションに拡張し
「このセクションは原本のどこから来たか」を常に確認可能にする。

**実装内容**:
- レビュー画面の ORIGINAL SOURCE ペインをタブ化: 「Text」(現行の originalText)と
  「Page image」(`sourcePage` に対応するページ画像。P0-1 のアセット配信 API を利用)。
- ページ画像はズーム(クリックでモーダル拡大)対応。
- セクションの `sourcePage` とページ画像の対応はサーバーの `normalizedDocument.pages` から
  引く。direct 経路など画像がない場合はタブを無効化し理由を表示。

**受け入れ基準**: サーバーパイプライン変換文書で、任意のセクション選択時に
該当ページのスナップショットを表示・拡大できる。

## F-3. 指示付き再生成(ヒューマンフィードバックループ)

**価値**: human-in-the-loop の中核強化。現在の Regenerate は同じ入力を再実行するだけで、
「矢印の向きが逆」「この表の 3 列目が欠けている」といった人間の観察を活かせない。

**実装内容**:
- レビュー画面の Regenerate をポップオーバー化し、自由記述の指示(任意)を入力できるようにする。
- `POST /api/documents/[id]/sections/[sectionId]/regenerate` と direct 用再生成 API の
  リクエストボディに `instruction?: string` を追加。
- `src/lib/llm/prompts.ts` の `buildSectionRegenerationPrompt` に
  「Reviewer instruction(最優先で従う。ただし原本にない事実の捏造は不可)」ブロックを追加。
- 指示履歴をセクションの `notes` に `[指示] ...` として残す(監査証跡)。

**受け入れ基準**: 指示を付けて再生成すると指示が反映され、notes に履歴が残る。
指示なし再生成は従来どおり動く。

## F-4. 複数ファイル一括変換(バッチ)

**価値**: 企業の実態は「章ごとのフォルダに数十ファイル」(`sample-inputs/` がまさにその形)。
1 ファイルずつの変換では実務に乗らない。

**実装内容**:
- アップロード画面で複数ファイル選択・フォルダドロップを受け付ける
  (`<input multiple>` + DataTransferItem の走査)。
- 変換キュー UI: ファイルごとに 待機 / 変換中 / 完了 / 失敗 を表示し、逐次
  (同時実行 1〜2)で変換 API を呼ぶ。完了分から順次 `/review/[id]` へ遷移できる。
- F-1 のダッシュボードにバッチ結果が並ぶことを確認する。
- サーバー側は既存 API の逐次呼び出しで成立させ、まずは新規のジョブキュー基盤を作らない。

**受け入れ基準**: 5 ファイルをドロップすると順に変換され、失敗ファイルはエラー表示のうえ
他のファイルの変換が継続する。

## F-5. Anthropic プロバイダ追加(プロバイダ非依存の実証)

**価値**: README は「特定 LLM プロバイダにハードコードしない」と明言しているが、
実プロバイダは OpenAI のみ。2 社目を実装して初めて抽象が本物になる。

**実装内容**:
- `src/lib/llm/anthropic-provider.ts` を新設。`@anthropic-ai/sdk` を追加し、
  Messages API + tool use(単一ツールに `review_document` スキーマを渡す方式)で
  `ReviewDocumentOutput` 相当の JSON を得る。既存の
  `review-document-schema.ts`(P1-2 で縮小後)と normalizer を再利用する。
- PDF は Anthropic のネイティブ document input、画像は image ブロックで渡す。
- `ConversionProviderName` に `"anthropic"` を追加し、`providers.ts`・convert ルートの
  provider バリデーション・`sample.env.local`(`ANTHROPIC_API_KEY`,
  `ANTHROPIC_MODEL`)を更新。モデル既定は `claude-sonnet-5`。
- アップロード画面のプロバイダトグルに追加(API キーがある環境のみ有効化)。

**受け入れ基準**: `DOCUTOR_LLM_PROVIDER=anthropic` で変換・再生成が通る。
プロバイダ固有コードが provider ファイルの外に漏れていない。

## F-6. エージェント向けエクスポート形式(RAG バンドル)

**価値**: 「Ready for agents」バッジ(complete 画面)を実体化する。単一 Markdown は
人間向けであり、エージェント/RAG パイプラインにはチャンク+メタデータが要る。

**実装内容**:
- ZIP エクスポートに `agent/` ディレクトリを追加:
  - `agent/sections.jsonl`: accepted セクションを 1 行 1 チャンクで出力。
    フィールド: `id`, `type`, `title`, `sourceFile`, `sourcePage`, `markdown`,
    `reviewStatus`, `notes`, 図なら `mermaid` / `drawioXml`。
  - `agent/document.json`: 文書メタデータ(タイトル、原本ファイル、変換日時、警告、
    セクション ID 順序)。
- `document.md` の先頭に YAML front-matter(title / source / generated / warnings 数)を追加。
- 実装場所は P1-3 で共通化したエクスポートモジュール。client / server 両経路で同一出力。

**受け入れ基準**: ZIP に上記ファイルが含まれ、`sections.jsonl` の各行が JSON として
パースでき、テストで構造が固定される。

## F-7. 日本語 UI(i18n)

**価値**: ターゲットは「伝統的な日本企業」。レビュー作業者が日常的に触る画面が
英語のみでは導入障壁になる。

**実装内容**:
- 依存を増やさない軽量方式: `src/lib/i18n/`(`ja.ts` / `en.ts` の辞書 + `useT()` フック +
  React Context)。ロケールは `navigator.language` 初期値 + ヘッダーのトグルで切替、
  localStorage に保存。
- 対象: アップロード / レビュー / 完了 / ダッシュボード画面の全 UI 文言、
  エラーメッセージ(API の `error` 文字列はキー化が大きくなるため、まず UI 層のみで可)。
- README.ja.md が既にあるため文言トーンはそれに合わせる。

**受け入れ基準**: 言語トグルで全画面が日本語/英語で切り替わり、リロード後も維持される。

## F-8. TODO / Unclear 品質パネル

**価値**: 「不明点を黙って埋めない」という Conversion Rules の運用面を完成させる。
現在 TODO/Unclear は complete 画面で件数表示のみで、どこにあるか探せない。

**実装内容**:
- レビュー画面に品質パネル(サイドバー下部 or ヘッダーからのドロワー)を追加:
  全セクションの `generatedMarkdown` / `notes` から `TODO:` / `Unclear:` を抽出し、
  セクション名+該当行を一覧表示。クリックで該当セクションへジャンプ。
- 未解決マーカーが残るセクションを Accept する際は確認を出す(ブロックはしない)。
- 抽出ロジックは純粋関数 `extractAttentionMarkers(document)` として `src/lib` に置き、
  complete 画面の件数計算もこれに置き換える(現在は正規表現の重複実装)。

**受け入れ基準**: マーカー一覧から該当セクションへ移動できる。ユニットテストで
抽出ロジックを固定する。

## F-9. ホスト環境向け永続ストレージ(検討タスクを含む)

**価値**: 現在のホスト版は localStorage 保存のため、ブラウザを変えると成果物が消え、
チームでレビューを分担できない。「組織のナレッジ資産化」には永続化と共有が必須。

**実装内容**(2 段階):
1. 設計スパイク: ストレージ抽象 `DocumentRepository` インターフェース
   (`get/save/list/delete`)を定義し、現行のファイルシステム実装と localStorage 実装を
   その背後に移す(P1-3 の共通化に接続)。
2. 実装: Vercel 環境向けに Vercel Blob(原本・アセット)+ KV/Postgres(job.json 相当)の
   リポジトリ実装を追加。認証は最初は「URL を知っている人がアクセス可能」な
   署名付きリンク方式で開始し、本格認証は別途判断。

**受け入れ基準**: self-hosted(FS)/ hosted(Blob)を環境変数で切り替えられ、
どちらでも同じ API で文書の保存・再開ができる。

## F-10. 大容量ファイル対応(ページ分割変換)

**価値**: ホスト版 4 MB / モデルコンテキストの制限で、実務の重い PPTX / PDF が扱えない。

**実装内容**:
- サーバーパイプライン側で、正規化済みページを N ページずつのウィンドウに分割して
  プロバイダを複数回呼び、セクション配列をページ順にマージする
  (`convert` ルート内のオーケストレーション + セクション ID の採番一意化)。
- 進捗をジョブ status に反映(`converting (3/10 pages)` のような詳細フィールド追加)。
- direct 経路は対象外(上限緩和はしない)。上限超過時のエラーメッセージで
  self-hosted 版のパイプラインを案内する。

**受け入れ基準**: 20 ページ超の PDF がサーバーパイプラインで変換でき、
セクションの `sourcePage` が正しいページを指す。
